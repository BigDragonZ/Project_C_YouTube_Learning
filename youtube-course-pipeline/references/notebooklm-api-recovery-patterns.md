# NotebookLM API Recovery Patterns

## Context

During batch processing of multiple courses (May 16, 2026), NotebookLM API exhibited persistent rate limiting and RPC failures. This document records the recovery patterns that worked.

## Error Patterns

### Pattern 1: Rate Limit During MOC/Anki Generation

**Symptom**:
```
Error: Chat request was rate limited or rejected by the API. Wait a few seconds and try again.
```

**Recovery**:
1. Wait 60-180 seconds
2. Retry the same `notebooklm ask` command
3. If still failing after 3 retries, proceed with manual content creation

**Manual MOC Template**:
```markdown
# {CourseName} 知识地图 (MOC)

## 课程概览
[1-2 sentence summary based on syllabus]

## 章节导航
[For each chapter in syllabus:]
### [[Ch_XX_Chapter_Title]]
- **核心命题**: [Thesis from syllabus]
- **关键词**: [Key terms]

## 跨章节关联
[Logical connections between chapters]

## 下一步学习建议
[3-4 actionable next steps]

---
*Generated by NotebookLM Pipeline*
*Signed: DALONG ZHANG*
```

**Manual Anki Template** (20 cards):
```markdown
# Anki - {CourseName} 真题卡 (20张)

## 卡片 1
**Q: [Core concept question]?**
A: [Detailed answer with derivation/formula]

[... repeat for 20 cards covering each chapter's key concepts ...]

---
*Generated by NotebookLM Pipeline*
*Signed: DALONG ZHANG*
```

### Pattern 2: RPC GET_LAST_CONVERSATION_ID Failed

**Symptom**:
```
RPC GET_LAST_CONVERSATION_ID failed after 1.4s
RPC GET_NOTEBOOK failed after 0.3s
```

**Recovery**:
1. This indicates the NotebookLM CLI's internal state is corrupted
2. The notebook itself is fine — the CLI just lost track of the conversation
3. Re-select the notebook: `notebooklm use <NOTEBOOK_ID>`
4. If that fails, the API is likely in a degraded state — wait 2-3 minutes
5. During degradation, manual content creation is the only viable path

### Pattern 3: Syllabus Generation Timeout in note_pipeline.py

**Symptom**:
```
RuntimeError: notebooklm failed after 3 attempts: RPC GET_NOTEBOOK failed
```

**Recovery**:
1. The notebook was created but the adapter can't find it
2. List notebooks: `notebooklm list | grep "CourseName"`
3. Use the notebook ID directly: `notebooklm use <ID>`
4. Generate syllabus manually via `notebooklm ask`
5. Format as `## 第N章：Title` for parser compatibility
6. Save to `01_Permanent/CourseName/CourseName_课程大纲.md`
7. Update checkpoint and resume

### Pattern 4: Empty Chapter Notes (All Rounds Failed)

**Symptom**:
- Chapter file size ~1.5KB (template only) vs ~15-25KB (successful)
- File contains: `[生成失败: notebooklm failed after 2 attempts: RPC GET_LAST_CONVERSATION_ID failed]`

**Recovery**:
1. Wait for API recovery (check with simple `notebooklm ask "hello"`)
2. Delete failed chapter files: `rm 01_Permanent/CourseName/Ch_*.md`
3. Reset checkpoint chapter statuses to `"pending"`
4. Re-run: `note_pipeline.py --course CourseName --resume --skip-upload`

**Prevention**: After Phase 2 completes, verify at least one chapter has substantial content (>10KB) before proceeding to Phase 3.

### Pattern 5: Source Upload Timeout (Partial Upload)

**Symptom**:
- `note_pipeline.py --phase 1` times out at 600s during upload
- Only partial files uploaded (e.g., 16/46)

**Recovery**:
1. Check actual upload count: `notebooklm source list | grep -c "ready"`
2. Upload remaining files via Python CLIAdapter or shell loop
3. Verify total count matches video count

**Python upload script**:
```python
from lib.adapters.cli import CLIAdapter
from pathlib import Path
import time

a = CLIAdapter()
a.use_notebook('<NOTEBOOK_ID>')
raw_dir = Path('flow/CourseName')
files = sorted(raw_dir.glob('*.md'))
existing = {s.get('name') or s.get('title', '') for s in a.list_sources()}

for f in files:
    if f.name in existing or f.name.startswith('_'):
        continue
    print(f'Uploading {f.name}...')
    a.add_source(str(f))
    time.sleep(1)

print(f'Total sources: {len(a.list_sources())}')
```

### Pattern 6: Notebook Deleted / Source Loss

**Symptom**:
- `notebooklm source list` returns empty
- Notebook ID in checkpoint no longer exists

**Recovery**:
1. Create new notebook: `notebooklm create "CourseName"`
2. Record new notebook ID
3. Re-upload all sources (see Pattern 5)
4. Update `.checkpoint.json` with new `notebook_id` and `phase1_done: true`
5. Resume from Phase 2: `note_pipeline.py --course CourseName --resume --skip-upload`

### Pattern 7: Multiple Notebooks with Same Name

**Symptom**:
- `notebooklm list` shows multiple notebooks with identical names
- `note_pipeline.py` fails with "No notebook found starting with 'X'" or uses wrong notebook

**Recovery**:
1. List all notebooks: `notebooklm list`
2. Identify which notebook has the sources: `notebooklm source list -n <ID>`
3. Delete duplicates: `notebooklm delete -n <ID> -y`
4. Update checkpoint with the correct notebook ID

## Prevention

- **Stagger course processing**: Don't process multiple courses simultaneously — API rate limits are per-account
- **Monitor API health**: If one course hits rate limits, expect the next to as well
- **Pre-build manual templates**: Have MOC and Anki templates ready for quick manual fallback
- **Validate early**: Check `notebooklm doctor` before starting batch operations
- **Use background mode**: For Phase 2 with 5+ chapters, always use `terminal(background=True)` to avoid 600s foreground timeout
- **Verify chapter content**: After Phase 2, check file sizes (>10KB = success, ~1.5KB = failure)
- **Cache source list**: Before bulk upload, get source list once to avoid N+1 API calls

## Session Log Reference

- MIS course: API recovered after ~5 minutes of rate limiting; manual MOC/Anki created
- Principles of Management (first attempt): RPC errors persisted; all chapter generation failed; manual fallback used for entire Phase 2/3
- Principles of Management (second attempt): API recovered after ~30 minutes; all 10 chapters successfully generated with full content; MOC and Anki auto-generated
- Key insight: Chapter notes (Phase 2) are the critical output. MOC and Anki can always be reconstructed from syllabus. Do NOT block on API recovery for non-critical synthesis steps.
